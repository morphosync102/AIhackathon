# ペルソナ先行 UI 設計

## 目的

作るものが決まった直後にペルソナを確認し、そのプロダクトに合う UI 方針を決める。事前固定のデザインを当てはめず、当日のテーマ、入力体験、対象ユーザーに合わせる。

## 使うタイミング

- プロダクト案が決まった直後。
- UI を作り始める前。
- カメラ、メモ画像、音声、URL、ファイルなどの入力体験が決まった後。
- `rapid-prototyping` や `demo-polish` の前段で UI 方針を固めたいとき。

## 最初に確認すること

1. 想定ユーザーは誰か。
2. その人はどんな場面で使うか。
3. UI はどんな印象にしたいか。
4. 避けたい雰囲気は何か。
5. 3 分デモで最初に伝えたい体験は何か。

## 決めること

- 画面のトーン。
- 色、余白、言葉づかいの方向。
- 最初の入力導線。
- AI 処理中の見せ方。
- 結果表示の形。
- 初見の審査員が最初に押すボタン。

## 基本方針

- 「AI にお願いする画面」ではなく、対象ユーザーが自然に使う道具に見せる。
- AI の存在は、便利な結果、判断理由、変換の質として伝える。
- 画面上の説明を増やすより、入力と結果の流れを短くする。
- モデル名、プロンプト、JSON、ログは前面に出さない。
- テーマとプロダクトに合うなら、かわいさ、信頼感、業務感、遊び心のどれを優先してもよい。

## 避けること

- ペルソナ未確認のまま UI を作る。
- 汎用チャットボット風の画面に逃げる。
- 黒背景、青紫グラデーション、未来感だけの AI 風 UI にする。
- 長いプロンプト入力欄を画面の中心に置く。
- 事前に決めた雰囲気を、当日のプロダクトに無理に当てはめる。

## 最終出力

当日の相談では、次の形で返す。

- ペルソナ。
- 利用場面。
- UI の印象。
- 避ける雰囲気。
- 入力導線。
- 結果表示。
- 3 分デモで最初に見せる体験。
