# カメラソリューション整形

## 目的

当日に出たカメラ系アイデアを、実装しやすく、初見で伝わり、AI 活用が見える形に素早く整える。

## 最初に決めること

1. カメラで何を見るのか。
2. 写真から AI が何を判断するのか。
3. ユーザーの次の行動がどう変わるのか。
4. 3 分デモで一番見せたい瞬間はどこか。

## 強い形

- 撮影前後で価値がはっきり変わる。
- AI が「見た」「分けた」「比べた」「気づいた」「提案した」ことが UI に出る。
- 結果が長文ではなく、スコア、分類、リスク、改善案、優先順位で見える。
- 審査員が説明なしで、撮る、解析する、結果を見る、まで進める。
- 失敗時でもサンプル画像やモック結果でデモを継続できる。

## 整形手順

1. アイデアを一文にする。
   例: 「カメラで [対象] を撮ると、AI が [判断] して、[次の行動] を出す」
2. 入力を最小にする。
   カメラ画像だけで足りない場合でも、補足テキストは 1 行までにする。
3. 出力を 3 つ以内の主要カードにする。
   推奨は「判定」「理由」「次の行動」。
4. AI の働きを UI に残す。
   検出した要素、判断理由、信頼度、比較、注意点のいずれかを表示する。
5. 実装リスクを下げる。
   カメラが不安定なら画像アップロード、サンプル画像、固定モックを同じ導線に入れる。

## 判断基準

- カメラを使う必然性があるか。
- 写真を見た瞬間、審査員が結果を予想したくなるか。
- AI なしでは面倒、難しい、または遅い判断になっているか。
- 3 分以内に価値が伝わるか。
- テーマが変わっても、対象と判断内容を差し替えれば使えるか。

## 避けること

- カメラを使うだけで、AI 判断が薄い案。
- 写真から返す結果がただの説明文だけの案。
- 撮影対象、判断、次の行動が曖昧な案。
- 精度検証やリアルタイム処理が必要すぎる案。
- テーマに合わせるためだけに無理にカメラを入れること。

## 最終出力

当日の相談では、次の形で返す。

- 一文コンセプト。
- カメラで見る対象。
- AI が判断すること。
- 画面構成。
- 3 分デモ手順。
- 実装を軽くする逃げ道。
