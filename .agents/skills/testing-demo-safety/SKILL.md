# デモ安全性テスト

## 目的

製作者の発表に頼らず、審査員が Chrome で提出 URL を触るだけで主要価値を理解できる状態にする。

## 使うタイミング

- 最初の縦の流れが動いた直後。
- 60 分時点で継続、磨き込み、別案を試すか判断するとき。
- 提出 URL を出す前。
- Gemini API やデプロイ設定を変更した後。

## 確認項目

1. 提出 URL がクリーンな Chrome またはシークレットウィンドウで開く。
2. サンプル入力から主要フローが最後まで完了する。
3. Gemini API が成功したとき、結果がカード、リスト、表、スコアなどで構造化表示される。
4. Gemini API が失敗したとき、ユーザーが次に何をすればよいか分かる。
5. ローディング、成功、エラー状態がそれぞれ見える。
6. ノート PC 幅とモバイル幅で、文字やボタンが重ならない。
7. API キー、環境変数、モデル内部ログ、プロンプト全文が画面やブラウザに露出していない。
8. 質疑応答がなくても、審査員が最初に押すボタンと価値を理解できる。

## UI とデモの確認

- 最初のボタンが明確か。
- サンプル入力が現実的か。
- ローディング中に不安にならないか。
- 結果の一番大事なカードがスクロールなし、または短いスクロールで見えるか。
- モバイル幅で文字が読めるか。

## AI API の確認

- Gemini API キーをサーバー側の環境変数から読んでいるか。
- 入力不足、キー不足、タイムアウト、API 失敗を制御されたエラーにしているか。
- AI の返答を UI が扱える形に正規化しているか。
- シークレットや生の認証情報をログに出していないか。

## 最終出力

提出前レビューでは、次を返す。

- デモ可否: 可、条件付き可、不可。
- 最も危ない失敗点。
- 提出前に直すべきこと最大 3 つ。
- 直さないこと。
