# AIhackathon

2 人で話し合いながら、1 人実装で進める AI ハッカソン準備リポジトリです。

このリポジトリには、共有ルール、戦略、プロンプトパターン、エージェント用スキル、運用チェックリストを置きます。実際のプロダクトは、イベントテーマ発表後に作ります。

方針や運用を変えた場合は、`AGENTS.md` と `README.md` も更新します。

## ブランチ

- `main`: 共有ドキュメント、ルール、戦略、プロンプトテンプレート、エージェント用スキル。
- `demo`: 1 人実装で統合し、提出 URL を作るブランチ。

他のブランチで作業する場合も、この Markdown 一式を参照できる状態にして開発します。

## 競技での優先事項

審査員は提出された Web アプリを Chrome で直接使います。製作者の発表に頼らず、3 分で価値が伝わるセルフデモを前提に作ります。質疑応答は補足であり、UI と結果表示が主役です。

1. 発表されたテーマに合っていること。
2. 動く URL を提出すること。
3. 挑戦性と独自性を示すこと。
4. プロダクト体験の中で AI 利用が見えること。

## 当日の流れ

| 時間 | 作業 |
| --- | --- |
| 0:00-0:15 | テーマを解釈し、アプリ種別を選び、一文の価値提案を書く。 |
| 0:15-0:30 | UI フロー、AI 出力、デモシナリオ、最初に作る縦の流れを固定する。 |
| 0:30-1:00 | 1 人実装で最低限動くものを完成させる。 |
| 1:00-1:15 | 2 人で継続、磨き込み、別案を試すか判断する。 |
| 1:15-2:20 | 選んだ方向で実装、改善、または別案の短期検証を進める。 |
| 2:20-2:45 | 提出候補を 1 つに絞り、デモ安全性と結果表示を磨く。 |
| 2:45-3:00 | 最終 URL 確認、バックアップスクリーンショット、提出。 |

## 進め方

- 実装者: UI、AI API、プロンプト、結果表示、デプロイまで 1 本の縦の流れを作ります。
- 相談相手: ペルソナ、発想、審査員目線、別案検討、デモ安全性を一緒に確認します。
- 30 分時点と 60 分時点で、作り続けるか、磨くか、別案を試すかを 2 人で決めます。

デモ URL が危ない場合は、別案探索を止めて提出候補の安定化を優先します。

## 参照するスキル

- 実装者: `rapid-prototyping`、`ai-api-integration`、`prompt-engineering`、`demo-polish` を主に使います。
- 相談相手: `persona-first-ui-design`、`hackathon-review`、`testing-demo-safety` を主に使います。
- カメラやメモ画像の案が出たら、`camera-solution-shaping` または `note-image-solution-shaping` を使います。

## ローカルセットアップメモ

テーマ発表前は、あえてスタックを固定しません。選定後に、スタック固有のセットアップ手順をここへ追記します。

```txt
パッケージマネージャー:
開発コマンド:
開発 URL:
デプロイ URL:
```

## 環境変数

AI API は原則 Gemini API を使います。選んだフレームワークに合わせてローカルの環境変数ファイルを作り、実際の API キーは絶対にコミットしないでください。
